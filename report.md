# Model Registry: Отчёт по проектированию и реализации

## 1. Анализ существующих проблем

Текущее состояние хранения моделей представляет собой набор директорий без какой-либо системы управления:

```
models/
models/mlds_1/my_model_v1/
models/mlds_3/sft_modell_v_123/
models/mlds_180/model_v1_v0_with_rank_dataset_0/
models/mlds_180/model_v1_v0_with_rank_dataset_1/
models/mlds_180/model_v1_v0_without_rank_dataset_0/
models/mlds_180/model_v1_v0_without_rank_dataset_1/
```

### Выявленные проблемы

| # | Проблема | Описание | Пример из текущей структуры |
|---|----------|----------|-----------------------------|
| 1 | **Отсутствие метаданных** | Нет информации о том, кто обучил модель, когда, на каких данных, с какими гиперпараметрами и какие метрики были получены. | Папка `my_model_v1` не содержит никакой информации кроме имени |
| 2 | **Хаотичное версионирование** | Нет единой системы версий. Каждая команда именует версии по-своему: `v1`, `v_123`, `v1_v0`. | `sft_modell_v_123` vs `model_v1_v0_with_rank_dataset_0` |
| 3 | **Отсутствие жизненного цикла** | Нет понятия стадий модели (разработка, тестирование, продакшен, архив). Невозможно понять, какая модель сейчас в продакшене. | Все модели лежат в одной структуре без разделения |
| 4 | **Нет поиска и обнаружения** | Невозможно быстро найти модель по критериям (метрики, дата, автор). Единственный способ — обход файловой системы. | Нужно знать точный путь к модели |
| 5 | **Отсутствие контроля доступа** | Любой пользователь может случайно удалить или перезаписать чужую модель. | Общая директория `models/` без разграничения прав |
| 6 | **Несогласованное именование** | Нет конвенции именования: опечатки, разные разделители, смешение информации о данных и версии в имени. | `sft_modell` (опечатка), `with_rank_dataset_0` (датасет в имени) |
| 7 | **Нет сравнения моделей** | Невозможно сравнить две версии модели по метрикам или параметрам. | — |
| 8 | **Риск потери данных** | Нет резервного копирования, проверки целостности (чексуммы), защиты от случайного удаления. | — |
| 9 | **Нет линковки с экспериментами** | Модель не связана с конкретным экспериментом, пайплайном обучения или датасетом. | Информация о датасете закодирована в имени папки, а не структурирована |

## 2. Требования к системе

### 2.1 Функциональные требования

**Управление моделями (CRUD):**
- FR-1: Регистрация новой модели (имя, описание, владелец)
- FR-2: Просмотр списка всех зарегистрированных моделей с пагинацией и фильтрацией
- FR-3: Обновление метаданных модели
- FR-4: Удаление модели (с каскадным удалением всех версий)

**Управление версиями:**
- FR-5: Создание новой версии модели с загрузкой артефактов (файлов)
- FR-6: Автоматическая нумерация версий (auto-increment в рамках модели)
- FR-7: Скачивание артефактов конкретной версии
- FR-8: Хранение метрик (accuracy, loss, F1 и пр.) в структурированном виде
- FR-9: Хранение параметров обучения (learning rate, epochs, batch size и пр.)
- FR-10: Вычисление и хранение контрольной суммы (SHA-256) артефактов

**Жизненный цикл:**
- FR-11: Управление стадиями версии модели: `development` → `staging` → `production` → `archived`
- FR-12: Получение текущей production-версии модели одним запросом
- FR-13: Ограничение: только одна версия модели может быть в стадии `production` одновременно

**Поиск и обнаружение:**
- FR-14: Поиск моделей по имени (подстрока)
- FR-15: Фильтрация версий по стадии
- FR-16: Теги (key-value) для категоризации моделей и версий

**Сравнение:**
- FR-17: Сравнение метрик двух версий одной модели

### 2.2 Нефункциональные требования

| # | Требование | Описание | Обоснование |
|---|-----------|----------|-------------|
| NFR-1 | **Простота развёртывания** | Система должна запускаться одной командой (`uv run fastapi dev src/main.py`) | ML-команды не должны тратить время на настройку инфраструктуры |
| NFR-2 | **REST API** | Все операции доступны через HTTP REST API с документацией OpenAPI | Универсальный интерфейс для интеграции с любыми инструментами и скриптами |
| NFR-3 | **Надёжность хранения** | Артефакты моделей хранятся с контрольными суммами, БД сохраняет данные на диск | Потеря обученной модели = потеря вычислительных ресурсов на переобучение |
| NFR-4 | **Производительность** | API должен отвечать < 200ms для операций с метаданными (без учёта загрузки/скачивания файлов) | Не должен замедлять пайплайны обучения |
| NFR-5 | **Расширяемость** | Хранилище артефактов абстрагировано (можно заменить локальный диск на S3) | По мере роста команд может потребоваться переход на облачное хранилище |
| NFR-6 | **Наблюдаемость** | Логирование всех операций | Аудит и отладка |

## 3. Архитектура системы

### 3.1 Обзор

Система построена по классической трёхслойной архитектуре:

```
┌─────────────────────────────────────────────────────┐
│                    Клиенты                          │
│  (curl / Python SDK / CI/CD / Web UI)               │
└──────────────────────┬──────────────────────────────┘
                       │ HTTP REST
                       ▼
┌─────────────────────────────────────────────────────┐
│                   API Gateway                       │
│               (FastAPI + Uvicorn)                   │
│                                                     │
│  ┌─────────────┐  ┌──────────────┐  ┌────────────┐  │
│  │ /models     │  │ /versions    │  │ /compare   │  │
│  │   CRUD      │  │   CRUD +     │  │   метрики  │  │
│  │             │  │   upload/    │  │            │  │
│  │             │  │   download   │  │            │  │
│  └──────┬──────┘  └──────┬───────┘  └─────┬──────┘  │
│         │                │                │         │
│  ┌──────▼────────────────▼────────────────▼──────┐  │
│  │              Service Layer                    │  │
│  │  (бизнес-логика, валидация, оркестрация)      │  │
│  └──────┬────────────────────────────┬───────────┘  │
│         │                            │              │
│  ┌──────▼──────────┐   ┌────────────▼────────────┐  │
│  │  Database Layer │   │   Storage Layer         │  │
│  │  (SQLAlchemy +  │   │   (абстракция хранилища)│  │
│  │   SQLite/PG)    │   │                         │  │
│  └──────┬──────────┘   └────────────┬────────────┘  │
└─────────┼───────────────────────────┼───────────────┘
          │                           │
          ▼                           ▼
   ┌──────────────┐         ┌──────────────────┐
   │  SQLite DB   │         │ Local Filesystem │
   │ (метаданные) │         │  (артефакты)     │
   └──────────────┘         └──────────────────┘
```

### 3.2 Обоснование выбора технологий

| Компонент | Технология | Обоснование | Альтернативы |
|-----------|-----------|-------------|--------------|
| **Web-фреймворк** | FastAPI | Автогенерация OpenAPI-документации, асинхронность, встроенная валидация через Pydantic, высокая производительность | Flask (нет async, нет авто-документации), Django (избыточен для API) |
| **ORM** | SQLAlchemy 2.0 | Стандарт де-факто для Python, поддержка миграций через Alembic, абстракция от конкретной СУБД | Tortoise ORM (менее зрелый), peewee (меньше возможностей) |
| **База данных** | SQLite (dev) / PostgreSQL (prod) | SQLite — zero-config для локальной разработки и небольших команд. PostgreSQL — для масштабирования. Переключение через строку подключения | MongoDB (не подходит: данные реляционные) |
| **Хранилище артефактов** | Локальная ФС с абстракцией | Простейшее решение для начала. Абстрактный интерфейс позволяет добавить S3 без изменения бизнес-логики | Сразу S3 (усложняет dev-setup) |
| **Менеджер пакетов** | uv | Быстрая установка зависимостей, воспроизводимость окружения через lock-файл | pip + venv (медленнее), poetry (тяжелее) |

### 3.3 Ключевые архитектурные решения

1. **Разделение метаданных и артефактов.** Метаданные (имя, версия, метрики, параметры) хранятся в реляционной БД для быстрого поиска и фильтрации. Артефакты (файлы моделей) хранятся в файловой системе/объектном хранилище. Это позволяет независимо масштабировать хранение данных и метаданных.

2. **Абстракция хранилища.** `StorageBackend` — абстрактный класс, реализации: `LocalStorageBackend`, (в будущем `S3StorageBackend`). Бизнес-логика не зависит от конкретного хранилища.

3. **Автоинкремент версий внутри модели.** Версия — это число, которое автоматически увеличивается. Это устраняет проблему хаотичного именования версий.

4. **Стадии жизненного цикла как enum.** Чёткий набор стадий (`development`, `staging`, `production`, `archived`) с валидацией допустимых переходов.

## 4. API и схема БД

### 4.1 Схема базы данных

```
┌───────────────────────────────┐       ┌──────────────────────────────────────────┐
│           models              │       │           model_versions                 │
├───────────────────────────────┤       ├──────────────────────────────────────────┤
│ id          INTEGER PK        │──┐    │ id              INTEGER PK               │
│ name        VARCHAR UNIQUE    │  │    │ model_id        INTEGER FK ──────────────┘
│ description TEXT              │  │    │ version         INTEGER                  │
│ created_by  VARCHAR           │  └───>│ description     TEXT                     │
│ created_at  TIMESTAMP         │       │ stage           ENUM(dev/stg/prod/arch)  │
│ updated_at  TIMESTAMP         │       │ artifact_path   VARCHAR                  │
│                               │       │ file_size       BIGINT                   │
└───────────────────────────────┘       │ checksum        VARCHAR(64)              │
                                        │ metrics         JSON                     │
                                        │ parameters      JSON                     │
                                        │ tags            JSON                     │
                                        │ created_by      VARCHAR                  │
                                        │ created_at      TIMESTAMP                │
                                        │ updated_at      TIMESTAMP                │
                                        │                                          │
                                        │ UNIQUE(model_id, version)                │
                                        └──────────────────────────────────────────┘
```

### 4.2 REST API

#### Модели

| Метод | Путь | Описание | Тело запроса | Ответ |
|-------|------|----------|-------------|-------|
| `POST` | `/api/v1/models` | Создать модель | `{name, description?, created_by}` | `201 Model` |
| `GET` | `/api/v1/models` | Список моделей | Query: `search?, limit?, offset?` | `200 [Model]` |
| `GET` | `/api/v1/models/{name}` | Получить модель | — | `200 Model` |
| `PATCH` | `/api/v1/models/{name}` | Обновить модель | `{description?}` | `200 Model` |
| `DELETE` | `/api/v1/models/{name}` | Удалить модель и все версии | — | `204` |

#### Версии моделей

| Метод | Путь | Описание | Тело запроса | Ответ |
|-------|------|----------|-------------|-------|
| `POST` | `/api/v1/models/{name}/versions` | Создать версию (multipart: файл + метаданные) | `file, description?, metrics?, parameters?, tags?, created_by` | `201 Version` |
| `GET` | `/api/v1/models/{name}/versions` | Список версий | Query: `stage?, limit?, offset?` | `200 [Version]` |
| `GET` | `/api/v1/models/{name}/versions/{version}` | Получить версию | — | `200 Version` |
| `GET` | `/api/v1/models/{name}/versions/{version}/download` | Скачать артефакт | — | `200 file` |
| `PATCH` | `/api/v1/models/{name}/versions/{version}/stage` | Изменить стадию | `{stage}` | `200 Version` |

#### Сравнение

| Метод | Путь | Описание | Тело запроса | Ответ |
|-------|------|----------|-------------|-------|
| `GET` | `/api/v1/models/{name}/compare?v1=X&v2=Y` | Сравнить две версии | — | `200 {version_1: {...}, version_2: {...}, metrics_diff: {...}}` |

### 4.3 Примеры запросов

**Регистрация модели:**
```bash
curl -X POST http://localhost:8000/api/v1/models \
  -H "Content-Type: application/json" \
  -d '{"name": "sft_model", "description": "SFT language model", "created_by": "mlds_3"}'
```

**Загрузка версии с метриками:**
```bash
curl -X POST http://localhost:8000/api/v1/models/sft_model/versions \
  -F "file=@model.pt" \
  -F 'description=Trained on rank dataset' \
  -F 'metrics={"accuracy": 0.95, "f1": 0.92}' \
  -F 'parameters={"lr": 0.001, "epochs": 10, "batch_size": 32}' \
  -F 'tags={"dataset": "rank_v2", "framework": "pytorch"}' \
  -F "created_by=mlds_180"
```

**Перевод в production:**
```bash
curl -X PATCH http://localhost:8000/api/v1/models/sft_model/versions/3/stage \
  -H "Content-Type: application/json" \
  -d '{"stage": "production"}'
```

**Получение production-версии:**
```bash
curl http://localhost:8000/api/v1/models/sft_model/versions?stage=production
```

**Сравнение версий:**
```bash
curl http://localhost:8000/api/v1/models/sft_model/compare?v1=2&v2=3
```
